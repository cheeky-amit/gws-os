# GWS OS Architecture

See the full design doc at `docs/design.md` for comprehensive details.

## Transport Layer

All Google API access goes through the `gws` CLI (google workspace CLI, Rust binary).

### Command Pattern
```
gws <service> <resource> <method> --params '<JSON>' [--json '<body>'] [--format json]
```

### Multi-Account
No native profile flag. Each account uses an isolated config directory:
```bash
GOOGLE_WORKSPACE_CLI_CONFIG_DIR="$HOME/.config/gws-profiles/$PROFILE" gws <command>
```

### Email Send
Requires base64url-encoded RFC 2822 message in the `raw` field:
```bash
gws gmail users messages send --params '{"userId":"me"}' --json '{"raw":"<base64url>"}'
```

## Memory System (Three-Tier CAR Architecture)

The memory system follows the Clustered Associative Recall (CAR) Protocol with three storage tiers:

### Tier 1: Raw Episodes
- **Location**: `memory/actions/*.jsonl`
- **Retention**: 7-30 days
- **Format**: Append-only JSONL, one event per line
- **Purpose**: Unprocessed action logs — every interaction is recorded here first

### Tier 2: Compressed Summaries
- **Location**: `memory/contacts/*.md`, `memory/topics/*.md`
- **Retention**: Months
- **Format**: Markdown with YAML frontmatter enriched with CAR metadata
- **CAR metadata fields**: `access_count`, `retrieval_cues`, `linked_contacts`, `streaks`, `amendments`, `status`
- **Purpose**: Distilled knowledge about contacts and topics, updated by consolidation from Tier 1

### Tier 3: Schemas/Patterns
- **Location**: `memory/graph.jsonl`
- **Retention**: Permanent
- **Format**: JSONL with weighted edges between nodes
- **Purpose**: Cross-cluster relationships and patterns — weighted edges connecting contacts, topics, actions, and accounts

## Graph Engine

Single Python script: `bin/gws-graph.py` with 5 subcommands:

| Subcommand | Purpose |
|------------|---------|
| `read` | Query nodes/edges with `--context-for`, `--limit`, `--sort` |
| `write` | Append edges/events to `graph.jsonl` |
| `compact` | Merge duplicate edges when graph exceeds 10K lines |
| `score` | Compute relevance scores for retrieval ranking |
| `consolidate` | Run tier promotion (Tier 1 to 2, Tier 2 to 3) |

**Relevance scoring formula**: `recency * frequency * connection * zeigarnik`
- **Recency**: Ebbinghaus decay curve with access reset (each access resets the decay timer)
- **Frequency**: `access_count` from CAR metadata
- **Connection**: Edge weight from graph (how connected a node is to the current context)
- **Zeigarnik**: Boost for incomplete/open items (unresolved threads, pending follow-ups)

**Concurrency**: File locking with `fcntl.flock()` on write operations. Atomic rename (`os.rename`) on compact to prevent partial reads.

## Consolidation Pipeline

| Cadence | Direction | What Happens |
|---------|-----------|-------------|
| **Daily** | Tier 1 to Tier 2 | Action logs consolidated into contact/topic node updates (access counts, streaks, amendments) |
| **Weekly** | Tier 2 to Tier 3 | Cross-cluster analysis generates graph edges, regenerates `memory/metamemory-index.json` |
| **Monthly** | Tier 3 review | Archive stale Tier 1 data, review Tier 3 edge weights, regenerate full metamemory |

Consolidation is triggered by `bin/gws-graph.py consolidate` (called by `/gws morning` and `/gws weekly`).

## Trust Progression

```
observe → suggest → assist → automate
```

- Per action type AND per contact
- Contact nodes are source of truth (not trust-levels.json)
- **ALL send/reply/schedule actions require user confirmation regardless of trust level**
- **Promotion**: streak >= threshold (configurable per trust level transition)
- **Demotion**: User override drops one level, resets streak counter, logs disagreement to Tier 1 action log
- Disagreement memory persists — repeated overrides on the same action+contact pair are tracked and influence future scoring

## Prospective Memory

Forward-planted triggers stored in `memory/prospective.jsonl`:
```json
{"id":"...","trigger":"contact:jane emails about Q2","action":"suggest scheduling review call","planted":"2026-03-25","expires":"2026-04-30","fired":false}
```

- **Checked at session start** (`/gws morning` preamble scans `prospective.jsonl`)
- **Fire when conditions match** — trigger text is matched against current context
- **Expire when stale** — entries past `expires` date are pruned during daily consolidation

## Metamemory

- **Location**: `memory/metamemory-index.json`
- **Purpose**: Knowledge inventory — what the system knows, how confident it is, and what it does not know
- **Generated by**: Weekly consolidation (`bin/gws-graph.py consolidate`)
- **Tracks**: coverage (which contacts/topics have rich data), confidence scores, known gaps (contacts with few observations), unanswered questions (patterns detected but not yet confirmed by user)

## Skill Execution Flow

1. Bash preamble runs → fetches headers, reads registry, reads memory nodes
2. Preamble output becomes context for Claude
3. Claude analyzes, categorizes, proposes actions
4. User confirms destructive/send actions
5. Post-action hook logs to memory
